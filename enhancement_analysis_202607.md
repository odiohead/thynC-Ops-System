### 참고 수치 (2026-07-17 기준)

- 운영 병원 190곳 / 프로젝트 242건 / 유지보수 누적 215건 (월 80건 추세)
- 유지보수 완료건 처리일: 보통 평균 4.3일(최대 34일), 높음 3.3일, 긴급 0일
- 미완료 유지보수: 접수 31 / 처리중 15 / 보류 3
- 장애유형: 하드웨어 80 > 기타 66 > 네트워크 46 > 소프트웨어 17
- 병원별 최다: 서울아산 19건, 원광대 10건, 한양대 9건

### 구조적 신호 세 가지

1. **유지보수가 시스템의 무게중심이 됐다.** 운영 병원 190곳, 월 80건 추세면 연 1,000건 규모.
   현재 유지보수는 "텍스트 기록 + 수동 상태" 구조 — 하드웨어 장애가 최다(80건)인데 **어떤
   장비(개체)가 고장났는지 구조화 기록이 없고**(증상 텍스트 내 '게이트웨이' 언급 14건뿐),
   처리시간 지표도 없다.
2. **데이터 정합성이 반복적으로 사람 손을 탄다.** 2026-07에만 병상 집계 정합화로 3세션 소요.
   현재도 상태 '완료'인데 `resolvedAt` NULL인 유지보수 **39건**, bed_count 미입력 완료
   프로젝트 3건 존재. 규칙은 알려져 있는데 감지가 수동.
3. **모듈 간 연결 고리가 이미 깔려 있는데 반대 방향이 비어 있다.** WMS 출고 전표는 유지보수에
   연결되지만(`workType=MAINTENANCE`), 유지보수 쪽에서는 장비·자재가 안 보인다. WMS 시리얼
   개체(설치 병원 `hospitalCode` 기록됨)가 활용을 기다리는 상태.

## 2. 추천안: 유지보수 2.0 (메인) + 데이터 정합성 자동 점검 (퀵윈)

### A. 유지보수 2.0 — 장비 이력·SLA·정기점검

#### Phase 1 — 장비 개체 연동 (WMS 투자 회수)
- 신규 조인 테이블 `MaintenanceUnit`(maintenanceId × unitId): 유지보수 등록/수정 시 해당
  병원에 **설치된 시리얼 개체**(InventoryUnit) 검색·선택 연결
- 유지보수 상세에 "관련 장비" 카드 + 교체 시 기존 출고·회수 전표 연결 표시
- 역방향 뷰: 인벤토리 개체 상세에 **장애 이력** 카드, 병원 상세 '사용 자재'에 장애 횟수 뱃지
  → "이 게이트웨이 3번째 고장" 판단 가능
- DB 1테이블 추가뿐, 기존 데이터 소급 불필요 (신규 건부터 축적)

#### Phase 2 — SLA 지표 대시보드 (`/maintenances/analytics`)
- 데이터는 이미 존재 (reportedAt / resolvedAt / statusChangedAt / priority).
  우선순위별 목표는 기존 지연 규칙(긴급1·높음3·보통7·낮음14일) 재사용
- 지표: 우선순위별 평균 처리일·SLA 준수율, 월별 접수/완료 추이, 장애유형×병원 매트릭스
  (다발 병원 식별), 담당자별 부하
- DB 변경 0 — 집계 API + Recharts 페이지만

#### Phase 3 — 정기점검(PM) 스케줄링 (사후 대응 → 예방 전환)
- `MaintenanceSchedule` 테이블: 병원별 반복 주기(분기/반기), 다음 예정일, 담당자,
  체크리스트 템플릿
- 스케줄러(기존 notify-scheduler 패턴)가 예정일 도래 시 유지보수 건 자동 생성
  (유형 '정기점검') + Slack 알림 + 간트차트 표시
- 190개 운영 병원의 점검 누락을 시스템이 방지

#### Phase 4 (선택) — 완료 품질 게이트
- 완료 상태 전환 시 resolvedAt·조치내용 필수화 (현재 39건 불일치의 재발 방지)

### B. 데이터 정합성 자동 점검 (퀵윈, 반나절~1일)

- `lib/integrity-rules.ts` 규칙 카탈로그 (초기 8종):
  1. 완료 프로젝트인데 bed_count 미입력
  2. 상태 '완료'인데 완료일(resolvedAt) NULL
  3. 운영 병원인데 완료 프로젝트 없음
  4. `hospitals.intro_beds` ≠ 완료 프로젝트 bed_count 합
  5. 담당자 없는 진행 중 업무
  6. 방문일정 없는 처리중 유지보수
  7. (추가 후보) 계약일 없는 운영 병원, HIRA 미연결 운영 병원 등
- `/api/integrity` + 메인 대시보드 카드(위반 N건 → 클릭 시 목록·해당 건 바로가기)
  + 주간 Slack 요약(기존 notify 인프라 재사용)
- 효과: 병상 정합화 같은 수동 조사 세션 소멸, sync-intro-beds 스크립트 실행 시점 안내

## 4. 제안 진행 순서

**B(정합성 점검) 먼저 → A Phase 1~3 순차.**
B는 즉효성 퀵윈, A는 데이터가 가리키는 성장 축.
확정 시 기존 컨벤션대로 `function_maintenance.md`(Phase·게이트·⏳결정필요 항목 포함) 설계
문서를 작성하고 Phase별 게이트 방식으로 진행.

### ⏳ 시작 전 사용자 결정 필요 항목

1. **정기점검 주기 정책** — 전 병원 일괄 주기 vs 병원별 지정
2. **SLA 목표** — 기존 지연 알림 규칙과 공유 vs 별도 설정 분리
3. **정합성 규칙의 자동 수정 여부** — 감지·보고만(권장) vs 일부 규칙 자동 보정
