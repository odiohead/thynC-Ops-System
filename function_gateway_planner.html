게이트웨이 배치 플래너 (Gateway Placement Planner) — 설계안

v0.2 · 2026-07-19 · thynC Operations System 신규 기능 · 상태: 설계 확정 중 — 샘플 도면 검토 완료, Phase 0 착수 대기

1. 개요

목적: 병원에서 전달받은 1개층 도면(스캔 PDF/JPG)을 업로드하면, AI가 도면의 공간 구조(복도·병실·화장실 등)를 인식하고 게이트웨이 커버리지 규칙에 따라 설치 위치 초안을 산출하여, 사람이 PowerPoint에서 검토·수정할 수 있는 편집 가능한 PPTX로 내려주는 도구.

설계 철학: AI는 80% 초안을 만들고, 최종 판단은 사람이 한다. 산출물의 빨간 점은 개별 도형 객체로 생성되어 PowerPoint에서 자유롭게 이동·삭제·복사할 수 있으며, 검토 완료된 슬라이드를 실제 설치계획 문서에 복사해 사용한다. 점 개수 = 장비 수량 = 견적 근거.

범위: 설치계획 문서 전체 생성이 아니라 도면 분석 + 게이트웨이 배치 표시 페이지 1장 생성에만 집중. 1회 처리 단위 = 1개층 도면 1장.

2. 확정 요구사항

항목내용
입력 형식스캔 PDF(구조정보 없는 이미지 PDF) 또는 JPG — 순수 이미지 분석. CAD 메타데이터 없음
축척축척 표기가 없어도 치수(길이) 표기는 일반적으로 존재 → 치수 기반 스케일 산출. 치수도 없는 도면은 후순위 처리
배치 규칙 (초기값) ① 게이트웨이 커버리지: 직경 10m
② 병실: 기본 2개, 매우 작은 병실은 1개
③ 공용화장실: 1개
④ 계단실·야외: 제외
⑤ 규칙은 추후 설정 화면에서 변경 가능해야 함
배치 위치운영관리시스템 내부에 개발하되, 별도 메인 네비게이션 메뉴로 완전 독립 도구로 사용 (병원/설치계획 연동 없음 — MVP 기준)
배치 대상 공간계단실·EV·야외 등 제외 목록 외에는 인식된 모든 실(미분류 포함)에 일단 배치 → 불필요한 곳은 사람이 PPT에서 점 삭제 (2026-07-19 확정)
산출물PPTX, A4 가로(29.7 × 21.0cm). 게이트웨이 = 빨간 점, 직경 0.2cm, 개별 도형 객체(이동·삭제 가능). 총 대수 표기. 점 번호 라벨(GW-01…)은 미적용 (2026-07-19 확정)
접근 권한ADMIN 이상 (네비 메뉴·API 모두. 추후 완화는 메뉴 관리 + 코드 권한 상수로 조정) (2026-07-19 확정)

아키텍처 결정 근거
  • 운영시스템 내부 개발 리스크 없음: 신규 라우트(/gateway-planner, /api/gateway-planner/*)와 신규 테이블만 추가 — 기존 모듈을 수정하지 않으므로 격리도가 높다. 인증(JWT)·S3·Anthropic API·감사로그 등 기존 인프라를 전부 재사용해 개발량이 크게 줄어든다.
  • 백그라운드 잡 패턴: AI 분석은 30초~2분 소요 예상 → HTTP 요청 안에서 처리하지 않고, 기존 심평원 연동(HiraSyncJob) 패턴(잡 테이블 + 비동기 처리 + 클라이언트 폴링)을 그대로 따른다. 검증된 패턴이라 신규 인프라(큐 서버 등) 불필요.
  • 배치 계산은 AI가 아니라 코드가 수행: AI는 "공간 인식(어디가 복도·병실·화장실인가)"까지만 담당하고, 점 좌표 산출은 결정론적 배치 엔진(TypeScript)이 수행한다. 규칙 변경 시 재분석 없이 재배치만 하면 되고, 결과가 재현 가능하며, AI 좌표 환각 리스크를 구조적으로 줄인다.

4. 처리 파이프라인

1업로드 & 래스터화
PDF/JPG 업로드(최대 30MB) → S3 원본 저장 → PDF는 poppler pdftoppm으로 200DPI PNG 변환(다중 페이지면 페이지 선택 UI), JPG는 그대로 → sharp로 장변 3000px 정규화 → 도면 방향 감지·회전 보정(good_2 사례: 세로 시트에 90° 회전 배치 — AI 방향 질의 + 사용자 회전 버튼 폴백) → S3 저장. 잡 상태 RASTERIZED
▼
2AI 공간 인식 (Claude Vision)
정규화 이미지를 Claude Vision에 전달. ① 치수 표기 판독 → 픽셀↔미터 스케일 후보 산출 ② 공간 세그먼트 추출 — 각 공간의 종류(복도/병실/공용화장실/계단/EV/야외/기타)와 bounding box(+가능 시 polygon)를 JSON으로 반환. 좌표 정확도를 위해 그리드 오버레이 기법 + 2차 검증 패스 사용(§8). 잡 상태 ANALYZED
▼
3스케일 확정 (사람 개입 지점 ①)
결과 화면에서 AI가 읽은 스케일 후보(예: "복도 치수 32,400 → 1px=0.021m")를 표시하고 사용자가 확인. 오류·부재 시 2점 클릭 보정: 도면에서 길이를 아는 두 점을 클릭하고 실제 거리(m) 입력 → 스케일 재계산. 스케일이 커버리지 간격 계산의 근거이므로 확정 없이는 배치 단계로 진행 불가
▼
4배치 엔진 (결정론적 코드)
공간 목록 + 스케일 + 배치 규칙(AppSetting)으로 좌표 산출 — 복도: 중심선을 따라 커버리지 직경 × 중첩계수 간격 배치 / 병실: 면적 기준 1~2개 / 공용화장실: 1개 / 계단·야외: 제외. 결과는 점 목록 JSON(공간 참조 포함). 잡 상태 PLACED
▼
5미리보기 & PPTX 생성 (사람 개입 지점 ②)
웹에서 도면 위에 점 오버레이 미리보기(공간 인식 결과도 색상 폴리곤으로 토글 표시) → "PPTX 생성" 클릭 → pptxgenjs로 도면 배경 + 빨간 점(개별 도형) + 총 대수 텍스트박스 슬라이드 생성 → S3 저장 + 다운로드. 잡 상태 DONE. 이후 최종 검토·수정은 PowerPoint에서
재실행 지원: 각 단계 결과를 잡 레코드에 저장하므로, 규칙 변경 후 "재배치"(4번부터), 스케일 수정 후 "재계산"(3번부터)을 AI 재호출 없이 수행 가능. AI 재분석(2번)만 토큰 비용 발생.

6. 데이터 모델

6.1 GatewayPlanJob (신규 테이블 — public.gateway_plan_jobs)

model GatewayPlanJob {
  id            Int       @id @default(autoincrement())
  title         String                       // 표시용 (기본: 원본 파일명)
  status        String                       // PENDING | RASTERIZING | ANALYZING | NEED_SCALE
                                             // | PLACED | GENERATING | DONE | ERROR
  originalKey   String                       // S3: 업로드 원본
  originalName  String
  pageIndex     Int       @default(0)        // PDF 다중 페이지 중 처리 페이지
  imageKey      String?                      // S3: 정규화 PNG
  imageWidth    Int?
  imageHeight   Int?

  scaleMPerPx   Float?                       // 확정 스케일 (m / px)
  scaleSource   String?                      // ai_dimension | manual_2point
  scaleMeta     Json?                        // AI 치수 판독 근거·2점 보정 입력값

  analysis      Json?                        // 공간 인식 결과 (spaces[] — §8.2 스키마)
  placements    Json?                        // 배치 결과 (points[] — 좌표·소속 공간·근거)
  gatewayCount  Int?
  rulesSnapshot Json?                        // 배치 시점의 규칙 (재현성)

  pptxKey       String?                      // S3: 생성된 PPTX
  errorMessage  String?
  tokenUsage    Json?                        // AI 호출 토큰 사용량 기록

  createdById   Int
  createdBy     User      @relation(fields: [createdById], references: [id])
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  @@index([createdById, createdAt(sort: Desc)])
  @@index([createdAt(sort: Desc)])
  @@map("gateway_plan_jobs")
}
  • 공간·점 데이터는 정규화 테이블 대신 JSONB 채택 — 잡 단위로만 조회하고 관계 질의가 없으므로 스키마 단순성이 이득. 위키 contentJson 전례.
  • 마이그레이션은 CLAUDE.md 규칙 준수: psql 직접 실행 + 수동 마이그레이션 파일 + migrate resolve --applied.

6.2 배치 규칙 (AppSetting — key: gw_planner_rules)

{
  "coverageDiameterM": 10,          // 게이트웨이 커버리지 직경 (m)
  "corridorOverlapFactor": 0.8,     // 복도 점 간격 = 직경 × 계수 (기본 8m — 음영 방지 중첩)
  "wardDefaultCount": 2,            // 병실 기본 설치 수
  "wardSmallCount": 1,              // 소형 병실 설치 수
  "wardSmallThresholdM2": 20,       // "매우 작은 병실" 면적 기준 (㎡) — 미확정 Q3
  "toiletCount": 1,                 // 공용화장실 설치 수
  "excludedSpaceTypes": ["stairs", "elevator", "outdoor"],
  "placeUnknownRooms": true,        // 미분류 실도 배치 (면적 기준 1~2개) — 사람이 PPT에서 삭제 (확정)
  "dot": { "diameterCm": 0.2, "color": "FF0000" }
}

8. AI 공간 인식 전략 (핵심 리스크 구간)

8.1 좌표 정확도 확보 기법

  • 그리드 오버레이: 정규화 이미지에 좌표 눈금(px 단위 보조선·라벨)을 합성해 전달 → 모델이 상대 위치가 아닌 눈금 기준으로 bounding box를 보고하게 함. Vision 모델의 좌표 환각을 크게 줄이는 검증된 기법.
  • 2-pass 검증: 1차 인식 결과(공간 폴리곤을 색상으로 오버레이한 이미지)를 다시 모델에 보여주고 "잘못 잡힌 공간·누락 공간"을 교정하는 검증 패스 1회.
  • 구조화 출력 강제: tool use(JSON schema)로 응답 형식을 고정 — 파싱 실패·형식 이탈 방지.
  • 대형 도면 타일링(후순위): 장변 3000px로 부족한 초대형 도면은 사분면 분할 분석 + 병합. MVP에서는 단일 이미지로 시작.

8.2 공간 인식 출력 스키마

{
  "dimensionReadings": [                    // 치수 판독 (스케일 후보 산출용)
    { "text": "32,400", "meters": 32.4, "fromPx": [120, 80], "toPx": [2850, 80], "confidence": "high" }
  ],
  "spaces": [
    {
      "id": "s1",
      "type": "corridor | ward | toilet | stairs | elevator | outdoor | other",
      "label": "AI가 읽은 실명 표기 (예: '5인실', 'T/R')",
      "bbox": [x1, y1, x2, y2],             // px, 정규화 이미지 기준
      "polygon": [[x,y], ...],              // 가능 시 (복도의 L/T자 형태 등)
      "confidence": "high | mid | low"      // low는 미리보기에서 강조 표시
    }
  ]
}

8.3 배치 엔진 규칙 (결정론적)

공간 유형배치 로직
복도 (corridor)폴리곤/박스의 중심선(장축) 산출 → 커버리지 직경 × 중첩계수(기본 8m) 간격으로 등분 배치. L/T자 복도는 세그먼트 분할 후 각 세그먼트 처리, 교차점 우선 배치. 세그먼트 길이가 간격 미만이어도 최소 1개
병실 (ward)면적(스케일로 ㎡ 환산) ≥ 기준값 → 2개(장축 1/4·3/4 지점), 미만 → 1개(중심)
공용화장실 (toilet)중심에 1개
계단·EV·야외제외 (미리보기에 회색 빗금 표시 — 제외됐음을 시각 확인)
기타/미분류 (other)배치함 — 면적 기준 병실과 동일 로직(1~2개). 불필요한 곳은 사람이 PPT에서 점 삭제 (2026-07-19 확정: "일단 다 표시하고 사람이 제거"). 미리보기에 "미분류" 뱃지로 구분 표시

9. PPTX 산출물 스펙

  • 슬라이드 크기: A4 가로 (29.7 × 21.0cm) — 설치계획 문서 규격과 일치 (2026-07-19 확정, pptxgenjs defineLayout 커스텀)
  • 구성:
    • 배경: 도면 이미지 1장 (비율 유지 최대 크기, 중앙 배치)
    • 게이트웨이: 빨간(#FF0000) 원형 도형, 직경 0.2cm, 각각 독립 shape 객체 — PowerPoint에서 개별 선택·이동·삭제·복사 가능. 번호 라벨 없음 (확정)
    • 우상단 텍스트박스: 게이트웨이 총 N대 (편집 가능한 일반 텍스트박스)
    • (후순위 옵션) 커버리지 반경 반투명 원
  • 좌표 변환: 이미지 px 좌표 → 슬라이드 상 이미지 배치 영역 기준 cm 좌표로 선형 변환. 도면 이미지와 점의 상대 위치가 픽셀 정확도로 유지됨
  • 파일명: GW배치_{잡제목}_{YYYYMMDD}.pptx

10. 개발 단계 (Phase)

Phase내용완료 판정 (게이트)
0. 프로토타입 검증 최우선확보된 샘플 4종(docs/gateway-planner-samples/)으로 ①Claude Vision 공간 인식 정확도 ②치수 판독·스케일 산출 정확도 ③배치 엔진 시제품 → 점 오버레이 이미지 산출까지 스크립트로만 검증 (UI 없음). 그리드 기법·프롬프트·모델 선정 실험. good 2종은 전 파이프라인, bad 2종은 인식 한계 실측샘플 도면에서 공간 유형 인식률·좌표 오차가 "사람 검토로 커버 가능한 수준"인지 사용자가 결과 이미지를 보고 판정. 여기서 막히면 설계 재검토 — UI 개발 착수 전 리스크 제거
1. 코어 파이프라인잡 테이블·업로드·래스터화·AI 분석·배치 엔진·PPTX 생성 — API 레벨 E2E도면 업로드 → PPTX 다운로드까지 API로 완주, 점이 PowerPoint에서 편집 가능
2. UI메인 페이지(업로드·목록)·잡 상세(폴링·스케일 확정·2점 보정·미리보기)·네비 메뉴 등록비개발자(사용자)가 화면만으로 전 과정 수행 가능
3. 규칙 설정화/settings/gateway-planner + 재배치 기능 + 규칙 스냅샷규칙 변경 → 재배치가 AI 재호출 없이 동작
4. 보강 (후순위)치수 없는 도면 대응(수동 스케일 전용 모드) · 다중 페이지 PDF 페이지 선택 · 대형 도면 타일링 · 웹 미리보기에서 점 편집 후 PPTX 반영 · 커버리지 원 표시필요 시 개별 진행
Phase 0을 강하게 권장: 이 기능의 성패는 "Claude Vision이 실제 병원 도면을 얼마나 잘 읽는가"에 달려 있고, 이는 실물 도면으로만 확인 가능. 반나절 수준의 스크립트 실험으로 최대 리스크를 먼저 제거한 뒤 UI에 투자하는 순서가 안전하다.
