# 티켓 시스템 — 설계 작업 계획 (설계안을 작성하기 위한 설계안)

> 이 문서는 **티켓 시스템의 설계안 자체가 아니라, 설계안을 어떤 순서·단위로 작성할지**를 정의하는 메타 문서입니다.
> 초대형 프로젝트이므로 설계 작업조차 컨텍스트 윈도우 한 번에 담기지 않는다는 전제에서 출발합니다.
> 모든 설계 세션은 이 문서를 먼저 읽고, 해당 설계 Phase의 산출물만 작성한 뒤 종료합니다.

---

## 사전 합의 사항 (2026-07-22~23 논의 확정)

설계 이전에 사용자와 이미 합의된 방향. 모든 설계 Phase는 이를 전제로 합니다.

1. **컨셉**: AWS 내부 티켓 문화(SIM/tt)를 사내 규모에 맞게 축소 도입 — "모든 업무는 티켓으로 시작되고, 큐에 배정되고, 이력이 티켓 안에 쌓인다"
2. **아키텍처 대원칙**: **티켓 = 공통 워크플로 껍데기(상태·배정·우선순위·이력·기한) / 도메인 레코드 = 구조화된 본문** — 1:1 연결
   - 워크플로 관점에서는 통합 개념, 데이터 관점에서는 기존 도메인 모듈(유지보수·업무·답사·설치계획 등) 유지
   - 목적: 티켓베이스 업무 문화 + 업무별 구조화 데이터 기반 인사이트·대시보드, **두 가지를 모두 확보**
3. **필드 소유권 규율**: 워크플로 정보는 티켓에만, 도메인 정보는 도메인 테이블에만 — 중복 저장 금지 (양쪽이 어긋나면 둘 다 못 믿게 됨)
4. **진행 방식**: 위키 개발과 동일한 문서 기반 Phase 방식 — 전역 결정(뼈대)은 한 세션에서 통으로 확정, 지역 결정(상세)은 해당 구현 Phase 직전에 그때그때 설계
5. **AWS 충실 재현 (2026-07-23 추가)**: 티켓 설계는 아마존 티켓시스템(SIM/tt)을 **따라할 수 있는 만큼 최대한 따라한다** — 큐(조직 단위 배정), CTI 3단계 분류, Severity 체계, correspondence/worklog, 상태 라이프사이클 등. D2a 옵션 시트에서 각 항목의 기본 권장안은 "AWS 방식 재현"이며, 축소·생략은 사내 규모상 불가피한 경우에만 근거를 달아 제안
6. **Severity 5단계 (2026-07-23 사용자 명시)**: 티켓 등급 분류는 AWS와 동일한 **Sev1~Sev5 5단계**로 한다 — 축소 금지

## 설계 Phase 로드맵

### D1 — 현황 인벤토리 (조사, 결정 없음) — 2스텝 분할

뼈대 설계의 입력 자료 수집. **이 Phase에서는 아무것도 결정하지 않습니다.**

#### D1a — 핵심 4모듈 인벤토리
1. 워크플로를 가진 기존 모듈 전수 **식별** (목록만): 최소 유지보수(maintenances), 업무(tasks), 답사(site-visits), 설치계획(install-plans) + 상태 전이가 있는 추가 모듈(예: 프로젝트, 운행일지, WMS 전표 등)
2. 핵심 4모듈(유지보수·업무·답사·설치계획)에 대해 표로 정리:
   - 상태 필드와 상태값 목록, 상태 전이 규칙(있다면)
   - 담당자/배정 필드 (개인 배정인지, 조직 단위인지)
   - 이력/로그 구조 (예: `maintenance_logs`)
   - 우선순위·기한 필드 유무
   - 알림/메일 연동 유무
   - 대략적 데이터 규모 (row 수)

**산출물**: `ticket_system_design.md` 생성 + "1. 현황 인벤토리" 섹션(핵심 4모듈 + 전체 모듈 목록)

#### D1b — 잔여 모듈 인벤토리 + 공통 요소 매트릭스
1. D1a에서 식별된 잔여 모듈에 대해 같은 항목 조사
2. 공통 요소 vs 모듈 고유 요소 매트릭스 작성 — "티켓 껍데기로 올릴 수 있는 것"의 후보 도출
3. 티켓으로 담기지 않는 업무 유형 확인 (도메인 레코드 없는 일반 요청성 업무 — tasks 모듈과의 관계가 쟁점이 될 것)

**산출물**: "1. 현황 인벤토리" 섹션 완성 + "1.x 공통 요소 매트릭스"

**게이트**: 사용자가 인벤토리를 확인하고 누락 모듈 없음을 승인 → D2 진행

### D2 — 뼈대 설계 (전역 결정) — 준비/결정 2스텝 분할

모든 구현 Phase가 공유하는 전역 결정. **결정 자체는 반드시 한 세션(D2b)에서 통으로** — 나눠서 결정하면 Phase 간 모순 발생 시 근거가 컨텍스트에 없음. 단, 결정의 *재료 준비*는 분리 가능.

#### D2a — 옵션·트레이드오프 시트 작성 (결정 없음)
아래 결정 필요 사항 각각에 대해 옵션 2~3개, 장단점, 권장안을 `ticket_system_design.md` "2. 전역 결정" 섹션에 **결정 보류 상태로** 정리. D1 인벤토리를 근거로 인용.

#### D2b — 결정 세션 (한 세션 통으로, 분할 금지)
D2a 시트를 놓고 사용자와 항목별 확정. 결정 내용 + 근거 1~2줄로 시트를 갱신.

**결정 필요 사항** (D1 결과에 따라 조정 가능)

| 항목 | 쟁점 |
|---|---|
| **티켓-도메인 연결 구조** | 폴리모픽 참조(ref_type + ref_id) vs 도메인 테이블에 ticket_id FK vs 연결 테이블. 도메인 레코드 없는 순수 티켓 허용 여부 |
| **tasks 모듈 처리** | 기존 tasks는 사실상 "일반 티켓"과 중복 — 티켓으로 대체? 도메인으로 존치? 이관? |
| **상태 라이프사이클** | 공통 상태 집합(예: 접수→배정→진행→대기→해결→종료)과 재오픈 규칙. 모듈별 기존 상태와의 매핑 방침 |
| **큐/배정 모델** | AWS식 큐(조직 단위 배정) 도입 여부·범위. 기존 Organization/Role과의 관계 |
| **분류 체계** | CTI(Category/Type/Item) 유사 분류 도입 여부와 단계 수. 분류→큐 자동 라우팅 여부 |
| **우선순위/심각도** | 단일 우선순위 vs AWS식 Severity. 기한(SLA) 필드 도입 수준 |
| **이력(worklog) 모델** | 공통 티켓 로그 테이블 신설 vs 기존 모듈별 로그 존치·연결. 에디터는 코딩 컨벤션상 Tiptap(HTML) 계열 유지 검토 |
| **DB 스키마 위치** | `public` vs 위키처럼 별도 스키마(예: `ticket`). 의존성 방향 규칙 |
| **기존 모듈 편입 순서** | 어느 모듈부터 티켓 껍데기를 씌울지, 병행 운영 기간의 이중 관리 방지책 |
| **알림** | 티켓 배정·상태 변경 시 알림 범위 (기존 알림 인프라 재사용 여부). 페이징류 실시간 알림은 도입하지 않음(규모 부적합) 전제 확인 |
| **대시보드 접점** | 티켓 레이어가 새로 제공할 프로세스 지표(소요시간·큐 체류시간·재오픈율 등)의 수집 방식 — 이벤트 기록 수준만 결정 |

**이 Phase에서 결정하지 않는 것** (구현 Phase 직전 상세 설계로 이연): 화면 레이아웃, API 시그니처, 마이그레이션 SQL, 컴포넌트 구조

**산출물**: `ticket_system_design.md`에 "2. 전역 결정" 섹션 작성 — 항목별 결정 내용과 근거 1~2줄

**게이트**: 위 결정 사항 전부에 대해 사용자 확정 → D3 진행. 위키의 "Phase 0 미확정 상태에서 Phase 1 코드 작성 금지"와 동일하게, **D2 미확정 상태에서 구현 코드 작성 금지**

---

### D3 — 구현 로드맵 작성

**작업 항목**
1. D2 결정을 구현 Phase로 분할 — 각 Phase는 "한 세션에서 완결 가능한 크기 + 단독 배포해도 시스템이 깨지지 않는 단위"로
2. Phase별 작업 항목 골자·검증 방법·게이트 정의 (상세 설계는 비워둠 — 각 Phase 시작 시 채움)
3. 기존 모듈 편입 Phase는 병행 운영·롤백 방안 포함

**산출물**: `ticket_dev_schedule.md` 생성 (위키 스케줄 형식 준수)

**게이트**: 사용자가 로드맵 승인 → 구현 Phase 1 진행 가능. 이후 진행은 `ticket_dev_schedule.md`가 마스터 문서가 되고, 이 문서(ticket_design_plan.md)의 역할은 종료

## 잠정 구현 단계 로드맵 (참고용 — D2 결정·D3에서 확정)

> ⚠️ 아래는 전체 그림 공유를 위한 **잠정안**입니다. 확정본은 D3 산출물인 `ticket_dev_schedule.md`이며, D2 결정에 따라 Phase 구성·순서가 바뀔 수 있습니다.
> 각 구현 Phase는 시작 시 **(a) 상세 설계 → 사용자 확인 → (b) 구현 → (c) 검증·게이트** 순서로 진행 (상세 설계와 구현을 같은 세션에서 해도 되는 소형 Phase는 D3에서 표기).

- **P1 — DB 뼈대**: 티켓·티켓로그·큐·분류 테이블 신설, Prisma 모델, 수동 마이그레이션 (UI 없음)
- **P2 — 티켓 코어 API**: 티켓 CRUD·상태 전이·배정·worklog API (화면 없음, API 레벨 검증)
- **P3 — 티켓 기본 UI**: 티켓 목록(큐 뷰)·상세·생성/수정 화면 — 최소 동작 확보
- **P4 — 순수 티켓 운영 개시**: 도메인 레코드 없는 일반 업무를 티켓으로 접수 시작. tasks 모듈 관계 처리(D2 결정 반영: 대체/존치/이관)
- **P5 — 도메인 편입 ①: 유지보수**: 유지보수에 티켓 껍데기 연결, 기존 데이터 백필, 병행 운영·롤백 방안 포함
- **P6 — 도메인 편입 ②: 답사**
- **P7 — 도메인 편입 ③: 설치계획**
- **P8 — 도메인 편입 ④: 잔여 모듈** (D1 인벤토리에서 편입 대상으로 확인된 것 — 필요 시 P8a/P8b 분할)
- **P9 — 알림 연동**: 배정·상태 변경 알림 (기존 알림 인프라 재사용)
- **P10 — 프로세스 지표·대시보드**: 소요시간·큐 체류시간·재오픈율 등 티켓 이벤트 기반 지표 수집 + 대시보드 화면
- **P11 — 안정화·마무리**: 전 모듈 티켓 흐름 점검, 문서(README 등) 정리, PROD 반영 준비

원칙: 도메인 편입(P5~P8)은 **모듈당 1 Phase** — 한 세션 크기 유지 + 모듈별 병행 운영·롤백 독립성 확보. PROD 반영은 매 Phase가 아니라 사용자가 요청하는 시점에 묶어서.

4. 세션 종료: 하단 체크리스트 갱신 + `DEV_HISTORY.md` 상단 기록
5. 빌드·git push는 사용자 명시 요청 시에만 (CLAUDE.md 절대 규칙)

**약속어**: "티켓 설계 Phase D‑N 진행해줘" / "티켓 설계 다음 단계로"
