1목표와 역할 정의 (3축)

외부 Flowise RAG 프록시에 의존하던 AI 어시스턴트를 시스템 내부에 직접 구현한 에이전트형 어시스턴트로 전면 재구축한다. 어시스턴트는 특정 병원 히스토리 조회 도구가 아니라, 회사 운영 데이터·제품 지식 전반을 답하는 범용 업무 어시스턴트다.

목표 5줄 요약

  1. 3축 질의 대응 — CS 지식(위키) · 단건 조회(운영 DB) · 집계(영업/운영 현황)를 도구(tool use)로 처리
  2. 상담이력 자산화 — 상담 내용을 병원별 병원 노트(위키 페이지)에 축적 → 다음 상담에서 어시스턴트가 재활용
  3. 대화 영속화 + 스트리밍 — 세션·메시지 DB 저장, SSE 실시간 응답
  4. 병원 컨텍스트 실동작 — UI에서 선택한 병원이 실제로 답변·조회에 반영
  5. Flowise 의존 제거 — 외부 EC2(43.201.26.9)와 관련 env 폐기
설계 원칙 — 신규 시스템이 아니라 기존 시스템에 대한 기능 추가다. 기존 패턴(인증·Prisma·위키 시스템 카테고리·설정 페이지)을 최대한 재사용하고 새 패턴 발명을 최소화한다. 특히 병원 노트는 프로젝트 이슈노트에서 이미 검증된 패턴을 그대로 복제한다.

6병원 노트 & 상담이력 축적

6.1 저장 전략 결정 — 벡터DB vs 병원 노트(위키)

A. 벡터DB (pgvector) 도입B. 병원 노트 — 위키 페이지 채택
구조상담이력을 임베딩해 유사도 검색병원 1:1 위키 페이지에 상담이력 append, 도구로 직접 조회
인프라pgvector 확장 + 임베딩 API + 인덱스 신규 구축추가 인프라 0 — 프로젝트 이슈노트 패턴·pg_trgm 검색 재사용
사람 접근성DB 안의 불투명 데이터 (사람이 못 읽음)위키에서 사람이 직접 읽고 편집·보완 가능 — 상담이력이 살아있는 문서 자산
검색 품질의미 검색 우수 (대량 데이터에서 유리)병원 단위로 이미 파티셔닝되어 있어 1차 요구엔 충분
판단1차는 B 채택. 병원당 상담이력 규모가 커져 노트 검색이 한계에 도달하면 pgvector를 Phase 확장으로 재검토 (그때도 병원 노트는 원본 저장소로 유지, 벡터는 인덱스 역할)

그림 3. 상담이력 축적 파이프라인 — 상담이 병원별 지식으로 쌓이고, 어시스턴트가 그 지식을 다시 사용하는 선순환
  • append 형식: ## 2026-07-18 상담 (홍길동) — 상담유형 헤딩 + 정제 본문 블록. 마크다운→BlockNote 블록 변환은 @blocknote/server-util 재사용 (이슈노트 이관 스크립트 선례)
  • 노트가 없으면 자동 생성 후 append (멱등 생성은 이슈노트 로직 재사용)
  • 병원 미지정(공통) 상담은 병원 노트 대상 아님 — 저장 생략 또는 consultation_queue(존치 시)로만

구현 리스크 — 협업 문서 append 정합성. 병원 노트가 협업 서버(Y.Doc)로 열려 있는 동안 서버가 content_json에 직접 append하면 협업 문서와 어긋날 수 있다. 구현 시 다음 순서로 처리: (1) 협업 서버에 해당 문서가 열려 있는지 확인 → (2) 열려 있으면 협업 서버 API(Hocuspocus)로 블록 삽입, 닫혀 있으면 content_json 직접 갱신(첫 열람 시 Y.Doc 시딩은 기존 동작) → (3) 버전 스냅샷·plain_text·백링크 갱신은 기존 위키 PUT 로직 재사용. 10보안·권한

  • 모든 신규 API는 기존 getAuthUser JWT 인증 필수. 세션은 소유자만 접근 (ADMIN도 타인 세션 열람 불가 — 삭제만 가능)
  • 도구는 화이트리스트 방식: 정의된 12종 외 어떤 DB 접근도 불가. 전부 Prisma SELECT (raw query 금지). Claude가 임의 SQL을 실행할 경로는 존재하지 않음
  • 조회 범위는 "로그인 사용자 전체 조회 가능"인 기존 시스템 관행과 동일 (병원·업무 데이터는 전 직원 열람 데이터)
  • 병원 노트 append는 위키 쓰기 권한(USER+) 준수, 감사로그 resource='wiki_page' UPDATE 기록 (기존 위키 관행)
  • ANTHROPIC_API_KEY는 서버 env — 현재 dev2는 placeholder이므로 실키 확보 필요 (미결 #1)
  • 도구 호출 기록은 ai_chat_messages.tool_calls로 사후 추적 가능. read-only 조회이므로 audit_logs에는 기록하지 않음
