1. 공통 규격

1.1 베이스 경로 · 요청 방식

  • 베이스 경로: {host}/mobiCAREConsole/API (각 표의 Path는 /API/... 형태로 표기하며, 실제 호출 시 앞에 /mobiCAREConsole 컨텍스트가 붙음)
  • 요청 방식: 조회·등록·수정·삭제 모두 POST + JSON 바디(@RequestBody) 관례. 파일 다운로드만 GET(경로 파라미터), 파일 업로드는 POST multipart/form-data(@ModelAttribute + MultipartFile).
  • 공통 요청 파라미터: 대부분의 ReqXxx DTO는 요청자 식별용 requester(요청 사용자 코드)와 organizationCode(조직 코드)를 공통 필드로 포함. 다운로드 GET 경로도 {requester}:{...} 형태로 요청자를 경로에 포함.

1.2 인증 (TokenInterceptor)

  • 인증 방식: 커스텀 액세스 토큰 기반. Spring Security는 세션 STATELESS + CSRF disable만 담당하고, 실제 요청별 인증은 TokenInterceptor.preHandle이 수행.
  • 토큰 추출: ReqJson.makeAccessToken(request) — 요청 헤더/파라미터에서 accessToken 추출. 토큰 전달 헤더는 SX-Auth-Token, x-auth-token, authorization. 유효성은 AccessTokenCacheService.isAvailableAccessToken(accessToken)로 캐시 검증.
  • 세션 갱신: /API/AccountUtil/SessionRenew, /API/Application/SessionRenew로 토큰(세션)을 재발급.
보안 주의 — 인증 우회 경로. 토큰 인터셉터에는 검사 없이 통과하는 우회 경로가 두 가지 존재합니다. 실제 문자열 값은 본 문서에서 노출하지 않습니다.
  • 시크릿 코드 우회: 요청 헤더 SX-Secure-Code(ReqJson.makeSecureCode)가 특정 고정 문자열(********)과 일치하면 토큰 검사 없이 통과.
  • 특수 토큰 우회: accessToken이 특정 고정 문자열(********)이면 무조건 통과(모니터링/헬스체크용 고정 토큰으로 추정).
두 우회 경로는 하드코딩된 고정 비밀값에 의존하므로, 값이 유출되면 전 API 인증을 우회할 수 있는 신뢰경계 리스크입니다.

1.3 인증 실패 응답

인증 실패 시 Content-Type: application/json, 바디 {"result": false, "error": 241, "message": <ErrorCode.getErrorMsg(241)>}(에러 241 = invalid accessToken). HTTP 상태는 200으로 반환되며, 성공/실패는 바디의 error 필드로 판별합니다.

1.5 CORS (WebSecurityConfig)

  • allowedOrigin=*, allowCredentials=true, maxAge=3600
  • 허용 헤더: authorization, content-type, x-auth-token, SX-Auth-Token, SX-Client-IP, SX-API-Route, SX-Secure-Code
  • 허용 메서드: OPTIONS, HEAD, GET, PUT, POST, DELETE, PATCH
  • /ws(WebSocket)는 permitAll, 세션 STATELESS, CSRF disable

1.6 공통 응답 구조 (ResJson)

  • 표준 성공/실패 래퍼는 ResJson 계열: { result: boolean, error: int, message: string, remoteIp: string, ... }.
  • 각 조회 API는 ResJson을 상속한 ResSelectXxx DTO로 데이터 필드를 추가.
  • 페이지네이션 응답은 ResSelectXxxList / ResSelectXxxPage DTO(리스트 + 페이지 정보).

1.7 공통 에러 처리 (ExceptionAdviceController — @RestControllerAdvice)

  • Exception.class → HTTP 500, ResJson 반환: error=500, message="Exception: <toString>, Class: <class>, Method: <method>, Line: <line>", remoteIp 포함. log.db.enabled=true이면 data_log 테이블에 ERROR 로그 적재(DataLogService.insert).
  • SQLException.class → HTTP 500(구현은 null 반환, "DataBase Error" 콘솔 출력).
  • 에러 코드 메시지는 net.seers.common.utils.ErrorCode.getErrorMsg(int)로 매핑.
에러코드의미
241invalid accessToken (인증 실패)
500서버 예외(ExceptionAdvice 래핑)

1.8 로깅 (DataLogAspect / DataLog)

AOP(DataLogAspect)로 컨트롤러 호출 시작/결과를 DataLog.makeMessage/makeResult로 기록. DataLogController(@ControllerAdvice)가 로그 조회 API를 제공.

1.9 ForManager / ForAdmin 접미사 관례

  • ...ForManager 메서드는 대부분 내부적으로 동명의 기본 메서드로 위임(relay). HttpServletResponse를 추가로 받아 매니저 서버(중앙 관리 콘솔)에서 온프레미스로 프록시하는 라우팅 계층(SX-API-Route 헤더 관련). 기능은 기본 API와 동일하되 호출 주체(권한 범위)가 다름.
  • ...ForAdmin은 상위 관리자(전체 조직) 스코프 조회/수정.
