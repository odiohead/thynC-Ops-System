1. 연동 아키텍처 개요

thynC 백엔드(Spring Boot 2.3.12, Java 8, 외장 Tomcat WAR)는 병원별 온프레미스로 배포되며, 활성 프로파일(spring.profiles.active) 한 줄로 병원별 연동 on/off·대상 주소가 결정됩니다. 모든 연동은 사내 모듈(module-emr / ack / cardio / skylabs / ezcaretech / his / push / stream)로 기능 경계가 분리되어 있고, 사용 여부는 application-thync-api-<프로파일>.properties 설정키로 토글됩니다.

병원 EMR/HISOracle HIS · HL7 IE · PACS
thynC 백엔드thync-api-onpremise 01.01.431
외부 서비스Cardio · SkyLabs · Firebase
수신(inbound)HIS 환자/의사/병동 조회, SkyLabs PPG/BP 패킷, ezcaretech 환자정보
송신(outbound)HL7 바이탈, DICOM ECG 리포트, EMR PDF, Cardio 홀터오더, FCM 푸시
연동의 5개 커스텀 축. 병원별 프로파일 차이는 사실상 (1) EMR 전송 on/off·기관코드, (2) HL7 TCP/SFTP 인터페이스, (3) DICOM PACS 전송 대상(AE/host/port), (4) Oracle HIS 직결, (5) 임상 정책(HR 소스·트렌드·알람)으로 압축됩니다. DB·Redis·디렉토리 등 인프라는 전 병원 동일(localhost 표준 구성)입니다.

① EMR 연동 개요

항목내용
목적측정 바이탈(체온·호흡·부정맥 이벤트)·ECG 리포트를 병원 EMR/HIS로 전달. 병원별로 전달 방식(HL7 / DICOM / Oracle 직결 / ezcaretech / PDF)이 상이하며, 본 항목은 그 공통 on/off 스위치를 정의
방식전송 대상(체온·호흡)은 큐 테이블(bio_signal_temp_emr_update, bio_signal_resp_emr_update)에 적재 후 스케줄러가 전송. ECG 스트립은 UploadEmrStrips로 업로드
프로토콜병원별 하위 연동(②~⑦, ⑩)에 위임
트리거·주기측정 세션의 measurement_info.send_interval(분)·auto_report 기준 주기 전송. 관리자 화면에서 연동 주기 설정(UpdateEmrPeriod)
관련 설정키emr.enabled (기본 false) · emr.organization.code (기본 ACK; 병원별 THYNC/BS/BJH/ICB/gil_hosp/MCH 등)
관련 APIPOST/API/Manager/SelectEmrData · /API/Manager/UpdateEmrPeriod · /API/Measurement/UploadEmrStrips
온·오프 조건emr.enabled=true인 병원 프로파일에서만 활성. 관리자 콘솔 기관정보의 HIS 연동 여부(organization.sync_his)·EMR 설정 화면과 연동

관련 DB: emr_interface_info(EMR 인터페이스 서버 IP·전송시각), bio_signal_temp_emr_update/bio_signal_resp_emr_update(업로드 큐/이력).

② HL7 v2.5 연동 (HAPI · ACK 모듈)

항목내용
목적병원 인터페이스 엔진(IE)으로 환자 바이탈·이벤트를 HL7 v2.5 메시지로 송수신 (ADT/ORU 계열) (메시지 유형은 추정)
방식사내 module-ack + HAPI 2.3(hapi-structures-v25) 파싱/생성. 수신은 Netty 기반 MLLP TCP 서버, 전송은 별도로 SFTP(jsch)로 ECG 리포트 파일 전달 가능
프로토콜HL7 v2.5 over MLLP(TCP) · 파일 전송은 SFTP/SSH
트리거·주기ack.tcp.server.sendInterval.fixed로 고정 주기 전송 여부 결정. 이벤트/측정 발생 시 송신 (추정)
관련 설정키ack.tcp.server.enable (기본 false) · ack.tcp.server.organization (INFINITT / ACK / HALLA / InitialAndParticularTime / PARTICULAR 중 병원 IE 규격) · ack.tcp.server.sendInterval.fixed · ack.sftp.enable (ECG 리포트 SFTP 전송) · tcp.server.enable(범용 TCP)
관련 API내부 TCP/스케줄러 처리(HTTP 엔드포인트 아님). 연동 대상 환자는 infinitt_patient(PACS 처방 TXT 임포트) 등으로 매핑
온·오프 조건ack.tcp.server.enable=true 병원(예: gg·hns 프로파일)에서 활성. organization 값이 병원 IE 벤더 포맷을 지정

③ DICOM PACS 전송 (dcm4che · ECG 리포트 C-STORE)

항목내용
목적ECG 측정 리포트(PDF)를 DICOM 오브젝트로 변환해 병원 PACS로 저장(영상 아카이브에 심전도 리포트 귀속)
방식사내 로직이 PDF → DICOM 변환 후 dcm4che 5.23.3 네트워크 모듈로 원격 AE에 C-STORE 전송. 변환/전송 결과는 dicom_transfer_log에 상태 기록
프로토콜DICOM DIMSE C-STORE (TCP)
트리거·주기리포트 생성 파이프라인(⑩) 완료 시 또는 관리 API 호출 시. 폴더 감시 후 변환→전송 (추정)
관련 설정키dicom.file.send.type (0=전송, 1=미전송) · dicom.remote.host · dicom.remote.port · dicom.remote.aetitle · dicom.remote.org(병원 조직코드) · dicom.file.patientid.prefix(환자ID 프리픽스 패딩)
관련 APIPOST/API/Admin/CreateDicomFolder · /API/Admin/DicomConvertAndSend · /API/Admin/DicomSend (인증 제외 /API/Admin 카테고리 — 내부/운영용)
온·오프 조건dicom.file.send.type=0 병원에서만 실제 전송. 대상(host/port/aetitle)은 병원별 상이 — 형식은 <PACS_IP>:<port>, AE=<대상 AE 타이틀> (예: 게이트웨이 AE / 수신 AE). 온프레 기본값(127.0.0.1:4242, AE ORTHANC)은 미전송 상태의 플레이스홀더

dicom_transfer_log.status: 0=none, 1=변환실패, 2=변환성공, 4=전송실패, 5=전송성공. 일부 병원은 대상 AE만 정의하고 send.type=1(미전송)로 두어 설정만 존재하는 상태(예: mch·gnasan).

④ Oracle HIS 직결 (ojdbc8)

항목내용
목적병원 HIS(Oracle) DB에 직접 접속해 재원 환자·의사·병동·진료과 정보를 조회하고, HIS 계정으로 SSO 로그인 처리
방식ojdbc8 19.3 + UCP로 Oracle 직결. SELECT 1 FROM DUAL로 커넥션 검증. 조회 결과는 his_patient/his_doctor/his_department/his_em_patient/rep_nurse 테이블로 동기화
프로토콜Oracle Net (JDBC over TCP, 서비스명/SID). 단일/RAC 노드 모두 지원
트리거·주기로그인·환자 목록 화면 진입 시 온디맨드 조회. 스케줄 동기화 병행 (추정)
관련 설정키oracle.datasource.driver-class-name=oracle.jdbc.OracleDriver · oracle.datasource.jdbc-url(병원별 host:port/service, RAC는 2노드) · oracle.datasource.username · oracle.datasource.password ********
관련 APIPOST/API/Account/LoginHIS · /API/Account/LogoutHIS · /API/Account/LoginHISforManager · /API/Doctor/SelectHisPatientList · /API/Doctor/SelectHisDoctorList · /API/Doctor/SelectHisStaywardList · /API/Doctor/SelectHisExinfo1~5 · /API/Doctor/SelectHisPacsinfo · /API/Util/HumcHISLogin
온·오프 조건oracle.datasource.*가 정의되고 enabled가 켜진 병원(예: bjh·bs·icb)에서 활성. 정의만 있고 enabled=false인 경우(예: believesewoong)는 미사용. 기관 마스터의 organization.sync_his로도 노출 제어

HIS 호출 실패 시 에러코드 1793 반환. 접속 정보(host/계정)는 병원별 상이하여 본 문서에는 형식만 기재.

⑤ mobiCARE Cardio 홀터 리포트 연동

항목내용
목적thynC ECG 데이터를 SEERS mobiCARE Cardio(홀터 분석 플랫폼)로 전송하고, 분석 완료된 홀터 리포트를 수신·조회

관련 APIPOST/API/CardioSync/selectCardioMeasurementCode · /insertECGData · /SendHolterOrderRequestCardio · /SelectCardioMeasurementInfoList · /InsertLinkInfoCardioUnlinked · /SendLinkInfoCardio · /DownloadHolterReport
온·오프 조건온프레 기본 켜짐(cardio.send.type=true). gg·hns·sev 등은 false로 비활성. 연동 계정/URL은 cardio_integration_setting(계정 ********·base_url)에 저장

관련 DB: link_cardio_info(전송 작업 단위·send_status), link_cardio_report_info(수신 리포트 파일), link_cardio_send_file_info, report_check.

⑥ SkyLabs CART-ON (반지형 혈압/PPG)

⑨ Kafka 생체신호 스트림 (biosignal.stream)

항목내용
목적대규모/클라우드 환경에서 게이트웨이 수신 생체신호를 Kafka 토픽으로 분산 처리(파싱 서버 ↔ 콘솔)
방식kafka-clients 2.5.1 + spring-kafka. 컨슈머가 biosignal.stream 토픽 구독. 온프레미스 표준 구성에서는 Netty 게이트웨이 직접 수신이 기본이며 Kafka 리스너는 비활성
프로토콜Kafka (TCP, lz4/snappy/zstd 코덱)
트리거·주기리스너 자동시작 여부에 따라 상시 컨슘. auto-offset-reset=latest
관련 설정키project.kafka.listener.sever.autostartup (온프레 기본 false) · biosignal.listener.topic (biosignal.stream, 클라우드는 .dev) · spring.kafka.bootstrap-servers · spring.kafka.consumer.group-id=myGroup
관련 APIHTTP 엔드포인트 없음(내부 컨슈머). 콘솔 실시간 전송은 WebSocket(STOMP) /topic/public/allLeads/bioSignalData/{measurementCode}
온·오프 조건autostartup=true(주로 클라우드 dev/test/premium)에서만 활성. 온프레 병원은 기본 off

⑩ PDF 리포트 변환·전송 파이프라인

항목내용
목적측정 결과 리포트를 PDF로 생성한 뒤 EMR/PACS/외부 플랫폼으로 전송하는 공통 파이프라인. ②HL7·③DICOM·⑤Cardio·⑦ezcaretech 전송의 파일 소스
방식PDFBox 2.0.27로 PDF 생성/변환. 폴더 상태 전이로 파이프라인 진행: report_download → converted(변환 성공) / converted_fail → transferred(전송 성공) / transferred_fail
프로토콜로컬 파일 폴더 감시 → 하위 연동 프로토콜(DICOM C-STORE / SFTP / REST)로 위임
트리거·주기자동 리포트(report_check.auto_report, 일별) 또는 측정 종료 시. measurement_info.send_interval(분) 주기
관련 설정키project.report.*(각 폴더 경로) · measurement_info.auto_report·send_interval
관련 APIPOST/API/Admin/DicomConvertAndSend(DICOM 경로) · /API/CardioSync/DownloadHolterReport · /API/EzCareTech/sendReport
온·오프 조건병원별 전송 대상(②~⑦) 활성 여부에 종속. 변환/전송 결과는 dicom_transfer_log(DICOM)·각 link_cardio_*에 상태 기록
