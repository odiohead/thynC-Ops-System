2. 백엔드(WAR) 배포

2.1 WAR 네이밍 규칙

산출물 파일명은 프로파일과 버전을 그대로 담습니다.

mobiCARE_Console_<프로파일>-<버전>.war
예) mobiCARE_Console_thync-api-onpremise-01.01.431.war

버전 정보는 MANIFEST.MF에 없고(=Manifest-Version: 1.0만 존재), 실제 버전의 원천은 application.properties의 ext.* 키입니다. 버전 확인 시 MANIFEST가 아니라 프로퍼티를 보십시오.

2.2 활성 프로파일 선택

17종의 설정 프로파일이 모두 WAR에 내장되어 있으며, 병원 선택은 application.properties 한 줄로 이뤄집니다.

# application.properties (WAR 내 WEB-INF/classes/)
spring.profiles.active=thync-api-onpremise    # 온프레 표준 템플릿
# spring.profiles.active=thync-api-onpremise-sev   # 세브란스(추정) 프로파일
# spring.profiles.active=thync-api-onpremise-bs    # 봉생(추정) 프로파일  ...

즉 병원별 WAR = active 프로파일만 다르게 지정/빌드한 것입니다. 프로파일 종류·차이는 별도 문서 「환경설정 정의서」 2·4절을 참조하세요.

2.3 Tomcat 배포

외장 Tomcat의 webapps/에 WAR를 배치하고 컨텍스트 경로를 확정합니다. ROOT 컨텍스트로 서비스하려면 ROOT.war로 리네이밍하는 방식이 일반적입니다 (추정 — 운영 컨텍스트명은 병원 톰캣 설정에 따름).

# 예시 흐름 (외장 tomcat9)
sudo systemctl stop tomcat9
sudo cp mobiCARE_Console_thync-api-onpremise-01.01.431.war \
        /var/lib/tomcat9/webapps/ROOT.war
sudo systemctl start tomcat9
# 로그로 기동 확인: /var/lib/tomcat9/logs/ 및 아래 7절 logback 경로

2.4 서버 포트·업로드 한계

설정키값의미
server.port80임베디드 기동 기준 포트. 외장 Tomcat 배포 시 Tomcat 커넥터 포트가 실제 서비스 포트

5. 데이터베이스 준비

JPA는 ddl-auto=none이므로 애플리케이션이 테이블을 생성하지 않습니다. 스키마·테이블은 DDL 스크립트로 사전 준비해야 합니다.

5.1 스키마 생성

-- MySQL 8, 스키마명은 thync 고정 (project.app.data.db.basic.table.schema=thync)
CREATE DATABASE thync CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;   -- (추정 문자셋)

5.2 DDL·초기 데이터 적재

스크립트역할
create_table (계열 스크립트)테이블·인덱스 등 스키마 구조 생성
insert_data (계열 스크립트)코드·기준정보 등 초기 데이터 적재

순서는 create_table → insert_data입니다. 병원 계정·권한은 「환경설정 정의서」 3절 DB 그룹을 따르며, 자격증명은 보안상 마스킹되어 있습니다.

5.3 샤딩 테이블 / 내년도 파티션 생성

생체신호 등 대용량 테이블은 샤딩/파티션 구조를 사용하며, 내년도 파티션·샤딩 테이블 생성은 Admin API를 통해 수행합니다 (추정 — 운영 시 연 1회 사전 생성 권고). 연말 전 미리 익년 파티션을 생성해 두어야 데이터 인입 실패를 예방할 수 있습니다.

6. 프론트엔드 배포

프론트엔드 콘솔 2종(서비스 콘솔·관리자 콘솔)은 빌드 과정이 없는 Vanilla JS SPA(seers frontframework)입니다. 웹서버의 ~/html/<DIRECTORY>/가 git 저장소이며, 배포 = git 강제 동기화입니다.

6.1 패키지 구조

index.html
├── js/ {common, controller, model, module, custom, util}
├── assets/
├── template/ {thync, console, seers, ...}   # 테마 디렉토리
└── manual/  thync_manual_rev.6_251126.pdf    # 사용자 매뉴얼 동봉 배포

PWA manifest(THYNC, standalone/landscape)를 포함합니다. WAR 내 static/에는 images/test.JPG 1개뿐으로, 백엔드는 정적 서빙에 사실상 미사용이며 FE는 별도 웹서버가 서빙합니다.

6.2 배포 산출물(tar)

6.3 배포 흐름 (git reset --hard 방식)

운영 PCdeploy-local.sh
ssh / sshpass웹서버 접속
웹서버deploy-server.sh
git 저장소html/<DIRECTORY>
단계스크립트동작
운영 PCdeploy-local.sh_deploy-local.config의 SSH_TYPE(FILE=키/ID=패스워드)·SSH_KEY/USER/PASS·SERVER_IP·TOKEN(git 토큰)·BRANCH·REMOTE·DIRECTORY를 읽어, ssh/sshpass로 웹서버 접속 후 html/$DIRECTORY에서 deploy-server.sh 실행
웹서버deploy-server.shexpect로 git 토큰 입력 자동화 → git fetch $REMOTE → git reset --hard $REMOTE/$BRANCH
주의 — 강제 동기화. 배포는 git reset --hard $REMOTE/$BRANCH로 웹서버의 로컬 수정을 전부 폐기하고 원격 브랜치 상태로 강제 정렬합니다. 웹서버에서 직접 수정한 파일은 배포 시 소실됩니다. 반드시 저장소에 반영 후 배포하십시오.

7. 로깅 구성

logback 설정은 2종이며, 프로파일의 logging.config로 택1합니다. 두 파일은 구조가 같고 LOG_PATH만 다릅니다.

파일LOG_PATH사용처
logback-console.xml/home/ubuntu/project/mobiCAREConsole/log/logback구형/클라우드 (dev·test·premium)
logback-console-ubuntu22.04.xml/var/lib/tomcat9/logs/logback온프레 기본 — Ubuntu 22.04 + tomcat9 systemd 샌드박스로 지정 폴더 외 쓰기 불가 → 경로 분리

공통 동작

  • 콘솔 appender(TEXT_CONSOLE): %date{Asia/Seoul} [%-5level] [%M] ----- %msg, root INFO로 콘솔만 활성
  • JSON 파일 appender(JSON_FILE): LogstashEncoder + 사내 데코레이터(PrettyPrintingDecorator / ISO8601DateDecorator), log.json → 일별 + 50MB 사이즈 롤링(log_yyyy-MM-dd.N.json), maxHistory 365일. 단, 정의만 있고 root에 미연결 상태
  • DB appender는 주석 처리 — 대신 앱 설정 log.db.enabled=true로 데이터 로그를 DB 저장
  • 프레임워크 로거(spring beans/jdbc/tx/orm/web, hibernate 등)는 WARN 고정
  • 우선순위: application.properties의 logging.level.*가 logback보다 우선. 기본 logging.level.net.seers=INFO, 일부 병원(bjh·cnuhh·gh·mch·gnasan)은 DEBUG
  • 시간대 Asia/Seoul 고정, 포맷 yyyy-MM-dd HH:mm:ss.SSS
  • 톰캣/GC 로그는 별도 log/tomcat8, log/gc 디렉토리 운용(make_directory 스크립트 기준)

8. 기동 및 점검

8.1 헬스체크

기동 후 HTTP 응답과 logback 로그로 정상 여부를 확인합니다. micrometer/management.metrics 및 JMX(9090)를 통한 지표 수집이 가능합니다 (운영 엔드포인트 노출 범위는 병원별 설정에 따름 — 추정).

8.2 점검 정지

정기 점검·긴급 개입 시 플래그 파일(4절)로 무중단 개입합니다.

# 점검 진입
touch config/checking_server_stop
# 게이트웨이 수신만 차단하고 싶을 때
touch config/gateway_receive_force_stop

8.3 메모리 정리

OS 페이지 캐시/메모리 정리는 memory_free.sh로 수행하며, 앱 설정 project.memory.clear.command가 이 스크립트 경로를 가리킵니다. 스케줄러/운영 정책에 따라 주기 실행합니다 (추정).

9. 업그레이드 / 패치
