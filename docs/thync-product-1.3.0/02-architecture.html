1. 아키텍처 개요 · 설계 원칙

thynC 백엔드는 단일 Spring Boot WAR로 패키징되지만, 내부적으로는 20종의 사내 모듈(module-*.jar)로 기능 경계가 분리된 모듈러 모놀리스 구조입니다. 프론트엔드(서비스·관리자 콘솔)는 빌드 없는 Vanilla JS SPA로, API 서버와는 REST + WebSocket으로만 결합됩니다.

설계 원칙구현 방식
온프레미스 우선, 클라우드 겸용동일 코드베이스에 프로파일 내장. infrastructure=onPremise/cloud 한 값으로 DB·저장소·캐시 대상 전환
기능 경계 모듈화도메인별 module-*.jar(his/emr/ack/cardio/skylabs/vcall 등)로 연동·기능 분리
고빈도 시계열 분산주차·해시·연 단위 샤딩으로 초당 다건 유입 데이터를 물리 테이블 분할
실시간성SockJS+STOMP WebSocket 파형 스트리밍 + 최신값 캐시 테이블
느슨한 FE 결합빌드 없는 SPA, REST(POST/JSON) 중심 + 토큰 헤더 인증
표준 기반 연동HL7 v2(HAPI)·DICOM(dcm4che)·Oracle HIS 직결로 병원 시스템 통합

2. 전체 구성도

현장 장비(웨어러블·게이트웨이)에서 시작해 API 서버·데이터 계층을 거쳐 콘솔로 이어지는 데이터 흐름과, API 서버가 병원 외부 시스템과 맺는 연동을 함께 나타냅니다.

3. 백엔드 계층 구조

백엔드는 전형적인 3계층(Controller → Service → Repository) 구조에 AOP 로깅과 인터셉터 인증이 횡단 관심사로 결합된 형태입니다. 데이터 접근은 Spring Data JPA + Hibernate(ddl-auto=none, MySQL5Dialect)로 수행합니다.

4. 프론트엔드 아키텍처

서비스·관리자 콘솔은 모두 빌드/번들링이 없는 Vanilla JS SPA(자체 "seers frontframework for vanilla")입니다. 정적 파일을 웹서버가 그대로 서빙하고, 런타임 로더가 URL을 해석해 필요한 컨트롤러 모듈을 동적으로 불러옵니다.

5. 실시간 스트리밍 아키텍처

실시간 파형·이벤트는 SockJS + STOMP over WebSocket(/ws, permitAll)로 콘솔에 전송됩니다. 게이트웨이 수신은 Netty TCP가 담당하고, 콘솔로의 브로드캐스트는 STOMP 토픽으로 분리됩니다.

게이트웨이Netty TCP 수신
수신·분석 파이프라인저장 · AI 부정맥 · 알람 판정
STOMP 브로커파형 토픽 · 이벤트 버스 토픽
콘솔(FE)SockJS+STOMP 구독
요소내용
수신 서버Netty 4.1.65 — 게이트웨이 생체신호 TCP 수신, HL7 MLLP TCP 서버 겸용
콘솔 전송Spring WebSocket/messaging 5.2.15 + SockJS(폴백) + STOMP
토픽 구성실시간 파형 토픽과 이벤트 버스 토픽(알람·이벤트) 분리 (추정: 파형/이벤트 채널 이원화)

6. 데이터 아키텍처 (샤딩)

고빈도 시계열 데이터를 단일 테이블 비대화 없이 처리하기 위해 3종 샤딩(주차/순번·해시·연 단위)을 병행합니다. 측정 세션 메타(measurement_info)의 shard(연도)·part(0~100) 컬럼이 각 세션의 샤드 위치를 라우팅합니다.

샤딩 유형명명 규칙대상 (종수 × 100)물리
주차/순번 샤드<name>_YYYY_NNNN (0000~0099)고빈도 시계열 14종 — activity_level·spo2·heart_rate·ecg_json_data·resp·resp_signal_data·ews·st_segment·pvc_count·ppg_raw_data·gr_ac_json_data·bio_signal_trend·bio_signal_summary_trend·analysis_arrhythmia1,400
해시 샤드<name>_00NN (0000~0099)파일 수신 상태 3종(측정코드당 1행, UNIQUE) — ecg_raw_data_info·ecg_json_data_info·gr_ac_data_info300
연 단위 샤드<name>_YYYYtemperature·device_status_info·bio_signal_event·arrhythmia_alarm·bio_signal_last_simple_data (5종)연 5
단일 테이블—조직·환자·장비·연동·로그 등 나머지147(논리)

주차/순번 샤드는 초당 다건 유입되는 파형·수치의 단일 테이블 비대화를 막고, 해시 샤드는 고빈도 UPDATE(수신 오프셋 갱신)를 분산합니다. 모든 시계열 샤드는 measurement_info.measurement_code를 FK(ON DELETE CASCADE)로 참조하므로, 측정 세션 삭제 시 관련 시계열이 일괄 삭제됩니다.

요소내용

7. 인증 · 보안 아키텍처

인증은 Spring Security가 아니라 커스텀 액세스 토큰 + TokenInterceptor가 담당합니다. Security는 세션 STATELESS·CSRF disable만 설정합니다. 토큰은 로그인(/Account/Login) 응답으로 발급되어 이후 모든 요청 헤더 SX-Auth-Token으로 전달되고, 캐시(AccessTokenCacheService)로 유효성이 검증됩니다.

요소내용
토큰 전달 헤더SX-Auth-Token (호환: x-auth-token, authorization)
검증AccessTokenCacheService.isAvailableAccessToken() — 캐시 기반
인증 실패HTTP 200 + 바디 {result:false, error:241, message:...} (에러 필드로 판별)
CORSallowedOrigin=*, allowCredentials=true. 허용 헤더에 SX-* 계열(Auth-Token/Client-IP/API-Route/Secure-Code)
WebSocket/ws permitAll(연결 후 동일 토큰 사용)
세션 갱신FE가 2시간 주기 /AccountUtil/SessionRenew 호출

인터셉터 제외 경로 (신뢰 경계 유의)

전 경로(/**)에 토큰 인터셉터를 적용하되, 아래 카테고리·경로는 토큰 검사에서 제외됩니다. 이는 장비/내부 호출을 위한 별도 신뢰 경계를 전제한 것으로, 네트워크 수준(방화벽·세그먼트)에서 보호되어야 합니다.

구분제외 대상
카테고리 전체/API/Admin/**, /API/Util/**, /API/Gateway/**, /API/Application/**, /API/Assignment/**, /API/Version/**, /API/Premium/**, /API/EzCareTech/**, /api/v1/**
개별 개방로그인·계정찾기(/Account/Login, /AccountUtil/Found*), /Server/SelectAPIServer, 일부 다운로드·알고리즘 조회 등
보안 유의. ① Admin/Util/Gateway/Application/Premium/EzCareTech 카테고리는 토큰 미검증이므로 내부망·장비망 격리가 전제되어야 합니다. ② 시크릿 코드(SX-Secure-Code) 및 특수 고정 토큰(헬스체크용)으로 토큰 검사를 우회하는 경로가 존재합니다. ③ 일부 온프레 프로파일에 AWS 액세스키·DB 자격증명이 평문 잔존합니다(값은 ******** 처리). 운영 시 자격증명 회수·경로 격리를 권고합니다.
